# Safe Code Generator

既存のアーキテクチャとパターンに従い、**安全にコードを生成する** スキル。
AIは「安全なコード生成アシスタント」として動作する。

## 基本原則

AIは以下を **決定しない**：
- アーキテクチャ設計
- モジュール境界
- ドメインモデル設計
- 大規模リファクタリング

AIは以下を **実行する**：
- 既存パターンに沿ったコード生成
- テスト雛形の生成
- ボイラープレートの生成

## 手順

### 1. 既存パターンの確認

実装対象と同じ層の既存コードを読み、パターンを把握する：

**Backend (Go)**
- 命名規則（ファイル名、構造体名、関数名）
- エラーハンドリングのパターン
- import の構成
- GORM の使い方
- chi ルーティングの書き方

**Frontend (TypeScript/React)**
- コンポーネントの構成（関数コンポーネント）
- hooks の使い方
- API呼び出しのパターン
- 型定義の方法

### 2. コード生成

既存パターンに **厳密に従って** コードを生成する。

#### Backend ルール

```go
// ハンドラの例 — 既存ファイルのパターンに合わせる
// chi のルーティングパターンに従う
// GORM の使い方は既存 repository を参考にする
// エラーハンドリングは既存パターンを踏襲する
```

#### Frontend ルール

```typescript
// 既存コンポーネントと同じ構成にする
// hooks は既存の hooks/ ディレクトリのパターンに従う
// API クライアントは既存の api/ ディレクトリのパターンに従う
```

### 3. テストコードの生成

実装と合わせてテストも生成する：

**Backend**
- `go test` で実行できるユニットテスト
- テーブルドリブンテストを使う（既存パターンに従う）

**Frontend**
- Vitest で実行できるテスト
- 既存テストのパターンに従う

### 4. 生成後の確認

コード生成後、以下を確認する：
- 既存パターンと一貫しているか
- 不要な依存を追加していないか
- セキュリティ上の問題がないか（SQLインジェクション、XSS等）

## 禁止事項

- 新しいライブラリやフレームワークを導入しない
- 既存のディレクトリ構造を変更しない
- 過度な抽象化をしない
- 必要以上のコメントやドキュメントを追加しない
- 変更していないコードに手を加えない

## 検証

コード生成後は必ず以下を実行する：

```bash
# Backend
make lint
make test

# Frontend
npm run lint
npm run typecheck
```

エラーがあれば修正してから完了とする。

## 注意

- 「既存パターンに従う」が最優先。パターンが見つからない場合は人間に確認する
- 1つのSliceの範囲内でのみコードを生成する。スコープを勝手に広げない
- セキュリティ脆弱性を絶対に入れない
