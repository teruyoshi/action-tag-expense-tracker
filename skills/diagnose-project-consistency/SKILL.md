# Project Consistency Diagnostic

リポジトリの構造・コマンド・テスト・ドキュメント・生成物の整合性を診断し、
AI駆動開発（AIDD）に適した状態かを評価するスキル。

AI開発では、構造の不整合がコード生成の精度低下に直結する。
このスキルは「AIのアーキテクト」として、構造崩壊を未然に防ぐ。

## なぜ必要か

通常のAI開発では、コード生成を繰り返すうちに構造が崩れていく。
このスキルで定期的に診断することで、以下のサイクルが回る：

```
AI → 診断 → 構造修正 → コード生成
```

診断なしだと：

```
AI → コード生成 → 構造崩壊
```

## 診断手順

以下の10カテゴリを順番に診断する。
各カテゴリで問題が見つかった場合、具体的なファイルパスと修正案を提示する。

---

### カテゴリ1: repo_map 整合性

repo_map（`repo_map.yaml` や `repo_map.md`）が実際のディレクトリ構造と一致しているかを検証する。

#### 検出ロジック

1. `repo_map.yaml`（または同等ファイル）を Read で読む
2. Glob で実際のディレクトリ構造を取得する
3. 以下を比較する：
   - repo_map に記載があるが、実際には存在しないパス → `PHANTOM_PATH`
   - 実際に存在するが、repo_map に記載がないパス → `MISSING_FROM_MAP`
   - パスは存在するが、階層が異なる（例: `backend/handler` vs `backend/internal/handler`） → `PATH_MISMATCH`

#### 判定基準

- 不一致が0件: 10/10
- 1〜2件: 8/10
- 3〜5件: 6/10
- 6件以上: 4/10

---

### カテゴリ2: Make コマンド粒度

頻出の開発操作が適切な粒度で Makefile にラップされているかを検証する。

#### 検出ロジック

1. `Makefile` を Read で読む
2. 以下の必須ターゲットが定義されているか確認する：
   - `dev` — 開発環境起動
   - `test` — テスト実行
   - `lint` — Lint実行
   - `build` — ビルド
3. 以下の推奨ターゲットも確認する：
   - `migrate` — DBマイグレーション
   - `e2e` — E2Eテスト
   - `clean` — クリーンアップ
4. `make all` のような粗い粒度のターゲットしかない場合は警告する

#### 判定基準

- 必須4つ + 推奨2つ以上: 10/10
- 必須4つ: 8/10
- 必須3つ: 6/10
- 必須2つ以下: 4/10

---

### カテゴリ3: 手打ちコマンド検知

`docker compose`、`go run`、`npm run`、`playwright test` などのコマンドが
Makefile や scripts にラップされずに直接使用されることを想定していないか検証する。

#### 検出ロジック

1. `CLAUDE.md`、`README.md`、`docs/` 内のドキュメントを Read で読む
2. 以下のコマンドパターンを Grep で検出する：
   - `docker compose` （Makefile 外での使用）
   - `go run`
   - `go build`
   - `npm run` （package.json の scripts 定義は除外）
   - `playwright test`
   - `npx`
3. これらが Makefile のターゲット内で使われている場合は OK
4. ドキュメントで「直接実行してください」と案内されている場合は `MANUAL_COMMAND_DETECTED`

#### 判定基準

- 検出0件: 10/10
- 1〜2件: 7/10
- 3件以上: 4/10

---

### カテゴリ4: 単体テスト不足検知

ビジネスロジックを含むファイルに対応するテストファイルが存在するかを検証する。

#### 検出ロジック

**Backend（Go）の場合：**

1. Glob で以下のパターンのファイルを検索する：
   - `backend/**/handler*.go`、`backend/**/service*.go`、`backend/**/repository*.go`
2. 各ファイルに対応する `*_test.go` が同じディレクトリに存在するか確認する
3. 存在しない場合 → `MISSING_UNIT_TEST`

**Frontend（TypeScript/React）の場合：**

1. Glob で以下を検索する：
   - `frontend/src/**/*.ts`、`frontend/src/**/*.tsx`
   - ただし `*.test.*`、`*.spec.*`、`*.d.ts` は除外
2. `hooks/`、`utils/`、`api/` 配下のファイルにテストがあるか確認する
3. 存在しない場合 → `MISSING_UNIT_TEST`

#### 判定基準

- カバー率 80%以上: 10/10
- 60〜79%: 7/10
- 40〜59%: 5/10
- 40%未満: 3/10

---

### カテゴリ5: ディレクトリ設計整合性

実際のディレクトリ構造が、CLAUDE.md や repo_map で定義された設計と一致しているかを検証する。

#### 検出ロジック

1. `CLAUDE.md` のディレクトリ構造セクションを Read で読む
2. 実際のディレクトリ構造と比較する
3. 以下を検出する：
   - 設計にあるが実際にないディレクトリ → `MISSING_DIRECTORY`
   - 設計にないが実際にあるディレクトリ → `UNEXPECTED_DIRECTORY`
   - 命名が異なる（例: `handler` vs `controller`） → `NAMING_MISMATCH`

#### 判定基準

- 不一致0件: 10/10
- 1〜2件: 8/10
- 3〜5件: 5/10
- 6件以上: 3/10

---

### カテゴリ6: 生成物重複検知

同じ概念を表すファイルが複数存在していないかを検証する。
AIDD では AI が似たファイルを複数生成してしまうことがよくある。

#### 検出ロジック

1. Glob でプロジェクトルート直下および `docs/` 配下のファイルを列挙する
2. 以下のパターンで重複候補を検出する：
   - ファイル名の類似性（例: `repo_map.md` と `repo_structure.md` と `project_map.md`）
   - 同じ拡張子で似た名前のファイル
3. 候補が見つかった場合、ファイル内容の先頭50行を比較する
4. 内容が類似している場合 → `DUPLICATE_ARTIFACT`

#### 重複候補のキーワードグループ

以下のキーワードが名前に含まれるファイル同士は重複候補：

- `map`, `structure`, `tree`, `layout`
- `guide`, `manual`, `howto`, `instructions`
- `config`, `settings`, `options`
- `spec`, `schema`, `definition`

#### 判定基準

- 重複0件: 10/10
- 1件: 7/10
- 2件以上: 4/10

---

### カテゴリ7: ドキュメント・スキル依存関係

スキルやドキュメントが参照しているファイル・スキルが実際に存在するかを検証する。

#### 検出ロジック

1. `skills/` 配下の全 `SKILL.md` を Read で読む
2. 各スキル内で参照されている他のスキル名・ファイルパスを抽出する
3. 参照先が実際に存在するか Glob で確認する
4. 存在しない場合 → `BROKEN_REFERENCE`
5. `CLAUDE.md` 内のスキル一覧と `skills/` の実際のディレクトリを比較する
6. 不一致がある場合 → `SKILL_LIST_MISMATCH`

#### 判定基準

- 壊れた参照0件 + リスト一致: 10/10
- 壊れた参照1〜2件: 7/10
- 壊れた参照3件以上またはリスト不一致: 4/10

---

### カテゴリ8: CI/CD 整合性

CI/CD設定が Makefile のターゲットと整合しているかを検証する。

#### 検出ロジック

1. `.github/workflows/`、`.gitlab-ci.yml`、`Jenkinsfile` などの CI 設定ファイルを探す
2. CI で実行されているコマンドを抽出する
3. 以下を確認する：
   - CI のステップが Makefile のターゲットを使っているか（直接 `go test` などを叩いていないか）
   - Makefile にあるテスト・Lint ターゲットが CI でも実行されているか
   - CI にあるが Makefile にないコマンドがないか
4. CI 設定ファイルが存在しない場合 → `NO_CI_CONFIG`（警告のみ）

#### 判定基準

- CI が Make 経由で全ターゲット実行: 10/10
- CI は存在するが一部直接コマンド: 7/10
- CI が Make を使っていない: 4/10
- CI 設定なし: N/A（警告）

---

### カテゴリ9: Lint / Formatter 整合性

Lint・Formatter の設定がプロジェクト全体で統一されているかを検証する。

#### 検出ロジック

1. 以下の設定ファイルの存在を確認する：
   - Backend: `.golangci.yml`（または `.golangci.yaml`、`.golangci.toml`）
   - Frontend: `.eslintrc.*`（または `eslint.config.*`）、`.prettierrc`（または `prettier.config.*`）
2. `CLAUDE.md` で言及されている Lint ツールと、実際の設定ファイルが一致するか確認する
3. `package.json` の `scripts` に lint/format コマンドが定義されているか確認する
4. 設定ファイルが存在しない場合 → `MISSING_LINT_CONFIG`
5. CLAUDE.md の記載と実際の設定が不一致 → `LINT_CONFIG_MISMATCH`

#### 判定基準

- 全設定ファイル存在 + CLAUDE.md と一致: 10/10
- 一部欠落: 7/10
- 大幅な不一致: 4/10

---

### カテゴリ10: Migration 整合性

DBマイグレーションファイルの命名規則・順序・整合性を検証する。

#### 検出ロジック

1. `backend/migrations/`（または同等ディレクトリ）配下のファイルを Glob で列挙する
2. 以下を確認する：
   - ファイル名が連番または日時ベースで一貫しているか
   - Up/Down のペアが揃っているか（使用ツールによる）
   - 最新のマイグレーションが models のスキーマ定義と矛盾していないか
3. 命名規則の不統一 → `MIGRATION_NAMING_INCONSISTENCY`
4. Up/Down ペア不足 → `MIGRATION_PAIR_MISSING`
5. マイグレーションディレクトリが存在しない場合 → `NO_MIGRATIONS`（DB使用プロジェクトの場合は警告）

#### 判定基準

- 命名一貫 + ペア完全: 10/10
- 軽微な不整合: 7/10
- 大きな不整合: 4/10
- マイグレーションなし（DB使用時）: 3/10

## 診断レポート出力フォーマット

全カテゴリの診断が完了したら、以下のフォーマットでレポートを出力する。

```
# PROJECT CONSISTENCY REPORT

プロジェクト: [プロジェクト名]
診断日時: [日時]
総合スコア: [平均スコア]/10

## スコアサマリー

| # | カテゴリ | スコア | 検出数 | 状態 |
|---|---------|--------|--------|------|
| 1 | repo_map 整合性 | 9/10 | 1件 | OK |
| 2 | Make コマンド粒度 | 8/10 | 0件 | OK |
| 3 | 手打ちコマンド検知 | 7/10 | 2件 | WARN |
| 4 | 単体テスト不足 | 5/10 | 4件 | NG |
| 5 | ディレクトリ設計整合性 | 10/10 | 0件 | OK |
| 6 | 生成物重複検知 | 10/10 | 0件 | OK |
| 7 | ドキュメント依存関係 | 7/10 | 2件 | WARN |
| 8 | CI/CD 整合性 | N/A | - | SKIP |
| 9 | Lint/Formatter 整合性 | 10/10 | 0件 | OK |
| 10 | Migration 整合性 | 7/10 | 1件 | WARN |

## 詳細

### [各カテゴリの詳細]

検出された問題:
- [ファイルパス]: [問題の種類] - [説明]

修正案:
- [具体的な修正手順]

## 優先対応事項

1. [最もスコアが低いカテゴリの修正案]
2. [次に低いカテゴリの修正案]
3. ...
```

## 状態の判定

- 8〜10/10: `OK` — 問題なし
- 5〜7/10: `WARN` — 対応推奨
- 1〜4/10: `NG` — 要対応
- 該当なし: `SKIP` — 対象外

## 使用するツール

- **Read**: `CLAUDE.md`、`repo_map.yaml`、`Makefile`、各設定ファイルの読み取り
- **Glob**: ディレクトリ構造の取得、ファイル存在確認
- **Grep**: コマンドパターンの検出、参照の抽出
- **Bash**: `make` コマンドの存在確認（`make -n` でドライラン）

## 注意事項

- このスキルは診断のみを行う。修正の実行は人間の承認を得てから行う
- スコアは目安であり、プロジェクトの特性に応じて重み付けを変えてよい
- CI/CD 設定がないプロジェクトではカテゴリ8はスキップする
- DB を使わないプロジェクトではカテゴリ10はスキップする
- 診断結果は必ず人間に報告し、修正の優先順位は人間が判断する
