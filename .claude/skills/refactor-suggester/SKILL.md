# Refactor Suggester

**小さく安全なリファクタリング** を提案するスキル。
大規模な設計変更ではなく、既存アーキテクチャ内での改善に限定する。

## 対象となるリファクタリング

- 関数の抽出（長すぎる関数の分割）
- 重複コードの統合
- 命名の改善
- 不要なコードの削除
- エラーハンドリングの改善
- 型の整理
- `repo_map.yaml` と実コードの乖離の修正

## 対象外（AIが判断しない）

- アーキテクチャの変更
- モジュール構成の変更
- ドメインモデルの再設計
- 複数モジュールにまたがる大規模リファクタリング
- 新しいライブラリやフレームワークの導入

## 手順

### 1. 対象コードの分析

指定されたファイルやモジュールを読み、以下を確認する：
- コードの複雑さ
- 重複の有無
- 命名の一貫性
- 不要なコード
- `repo_map.yaml` の記述が実コードと一致しているか

### 2. リファクタリング候補の抽出

改善可能な箇所を列挙する。各候補について：
- 現状の問題点
- 提案する改善内容
- 改善による効果
- リスク（影響範囲）

### 3. 優先順位付け

以下の基準で優先順位をつける：
- **高**: バグの温床になりうる問題（エラー未処理、型の不整合）
- **中**: 可読性・保守性の改善（重複排除、命名改善）
- **低**: スタイルの改善（フォーマット、コメント）

### 4. 安全性の確認

各リファクタリングについて：
- 動作が変わらないことの確認方法
- 影響を受けるテスト
- ロールバックの容易さ

## 出力フォーマット

```
## リファクタリング提案

### 1. [提案名] (優先度: 高/中/低)

**現状**: [問題の説明]
**提案**: [改善内容]
**効果**: [改善後のメリット]
**リスク**: [影響範囲]
**検証**: [動作確認方法]

### 2. ...
```

## リファクタリングのサイズ基準

1つのリファクタリングは：
- 1〜3ファイルの変更
- 動作を変えない
- 独立して検証可能

これを超える場合は、さらに分割するか、人間と相談する。

## 検証

リファクタリング適用後：

```bash
make quick-check   # lint + lint-frontend + typecheck
make check         # quick-check + test + test-frontend
```

全て通ることを確認してから完了とする。

## プロジェクト状態の参照

リファクタリング分析時は、同ディレクトリの `PROJECT_STATE.md` を参照すること。
確認済みの状態が記録されており、再分析の起点となる。
分析後は `PROJECT_STATE.md` を更新すること。

## 注意

- リファクタリングは提案のみ。実行は人間の承認後
- 「改善のための改善」は避ける。明確な理由がある場合のみ提案する
- 既存のテストが全て通ることが前提条件
